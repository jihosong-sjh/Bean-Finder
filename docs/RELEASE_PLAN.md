- 개인정보 처리방침 법무 검토
- 유료 광고 집행

## 2. 릴리즈 목표

MVP 베타 릴리즈의 목표는 사용자가 다음 핵심 흐름을 문제없이 수행할 수 있는지 검증하는 것이다.

- 취향 기반 원두 검색
- 필터와 정렬을 통한 후보 좁히기
- 원두 상세 정보 확인
- 최대 4개 원두 비교
- 카테고리와 랭킹 탐색
- 외부 판매처 이동

이번 릴리즈는 자체 결제나 회원 기능이 없는 검색/비교 중심 MVP다.

## 3. 릴리즈 전략

### 3.1 기본 전략

MVP는 프론트 중심 배포를 기본으로 한다.

```text
Frontend App
  + Serverless/Internal API
  + JSON Data
  + Static Assets
```

별도 운영 DB와 독립 백엔드 서버는 사용하지 않는다. 단, `/api/v1/*` contract는 유지해 이후 DB 도입 시 프론트엔드 변경을 최소화한다.

### 3.2 배포 방식

권장 방식:

- Vercel, Netlify, Cloudflare Pages 등 프론트 배포 플랫폼
- 필요 시 같은 플랫폼의 서버리스 API 사용
- JSON 데이터는 앱 빌드 또는 서버리스 런타임에서 참조

### 3.3 릴리즈 유형

| 유형 | 설명 |
| --- | --- |
| internal alpha | 개발자와 기획자 내부 검증 |
| closed beta | 제한된 사용자에게 URL 공유 |
| public beta | 공개 접근 허용, 데이터와 지표 수집 |
| v1.0 | 데이터 확장과 운영 프로세스 확정 후 정식 공개 |

이 문서는 `closed beta` 또는 `public beta` 수준의 MVP 릴리즈를 기준으로 한다.

## 4. 환경 정의

| 환경 | 목적 | 데이터 | 접근 |
| --- | --- | --- | --- |
| local | 개발과 단위 검증 | 샘플 데이터 30-50개 | 개발자 |
| staging | 출시 전 QA | 실제에 가까운 데이터 300개 이상 | 내부/제한 사용자 |
| production | 실제 사용자 공개 | 운영 데이터 | 공개 |

### 4.1 local

목적:

- 개발 중 기능 검증
- 데이터 검증
- 단위/API/E2E 테스트 실행

완료 기준:

- 개발 서버 실행 가능
- 데이터 검증 통과
- 핵심 테스트 통과

### 4.2 staging

목적:

- 실제 배포 환경에 가까운 조건에서 QA
- 모바일/데스크톱 확인
- 링크, 이벤트, 성능 확인

완료 기준:

- production과 동일한 빌드 방식 사용
- staging URL에서 핵심 사용자 흐름 통과
- 출시 차단 이슈 없음

### 4.3 production

목적:

- 실제 사용자에게 공개
- 검색/필터/상세/비교 사용 지표 수집
- 데이터 품질과 UX 검증

완료 기준:

- 배포 성공
- smoke test 통과
- 오류율과 핵심 이벤트 수집 확인

## 7. 배포 전 체크리스트

### 7.1 코드와 빌드

- 타입 검사가 통과한다.
- lint가 통과한다.
- 빌드가 성공한다.
- 사용하지 않는 debug UI가 제거되어 있다.
- production 환경 변수가 설정되어 있다.
- 내부 검증 API가 공개되면 안 되는 경우 비활성화되어 있다.

### 7.2 데이터

- 원두 데이터 검증이 통과한다.
- 원두 300개 이상이 준비되어 있다, closed beta는 30-50개도 허용한다.
- 모든 원두에 가격과 용량이 있다.
- 모든 원두에 100g당 가격이 계산된다.
- 모든 원두에 판매처 URL이 있다.
- 품절 원두는 `is_available=false`로 표시된다.
- `last_checked_at`이 미래 날짜인 데이터가 없다.
- 90일 이상 오래된 데이터는 known issue 또는 갱신 대상으로 기록된다.

### 7.3 기능

- 홈 화면에서 검색할 수 있다.
- 검색 결과 목록이 표시된다.
- 필터가 동작한다.
- 정렬이 동작한다.
- 원두 상세 화면으로 이동할 수 있다.
- 비교함 추가/제거가 동작한다.
- 카테고리 화면이 동작한다.
- 랭킹 화면이 동작한다.
- 판매처 이동이 새 탭으로 열린다.
- 빈 결과 상태가 표시된다.

### 7.4 이벤트

- `search_submitted`가 기록된다.
- `filter_changed`가 기록된다.
- `bean_detail_viewed`가 기록된다.
- `compare_added`와 `compare_removed`가 기록된다.
- `outbound_clicked`가 기록된다.
- 이벤트 실패가 사용자 동작을 막지 않는다.

### 7.5 접근성과 반응형

- 모바일 390x844에서 핵심 화면이 깨지지 않는다.
- 태블릿 768x1024에서 핵심 화면이 깨지지 않는다.
- 데스크톱 1440x900에서 핵심 화면이 깨지지 않는다.
- 키보드로 검색, 필터, 정렬, 비교함을 조작할 수 있다.
- 맛 점수는 색상 외 숫자와 라벨로도 확인 가능하다.
- 이미지 대체 텍스트가 제공된다.

## 8. 배포 전 명령

실제 명령은 구현 스택 확정 후 `README.md`에 기록한다. 릴리즈 관점의 최소 실행 순서는 다음과 같다.

```text
install dependencies
type check
lint
data validation
unit tests
api tests
core e2e tests
build
```

권장 npm script 예시:

```text
npm run typecheck
npm run lint
npm run validate:data
npm run test
npm run test:api
npm run test:e2e
npm run build
```

## 9. Staging 배포 절차

1. 릴리즈 대상 branch 또는 commit을 확정한다.
2. 데이터 파일이 최신인지 확인한다.
3. 배포 전 명령을 local 또는 CI에서 실행한다.
4. staging 환경으로 배포한다.
5. staging URL을 확인한다.
6. smoke test를 실행한다.
7. QA 체크리스트를 실행한다.
8. 발견된 이슈를 severity 기준으로 분류한다.
9. 출시 차단 이슈가 있으면 수정 후 다시 staging 배포한다.
10. 출시 차단 이슈가 없으면 production 배포 승인을 요청한다.

## 11. QA 승인 기준

### 11.1 승인 조건

production 배포 전 다음을 만족해야 한다.

- `TEST_PLAN.md`의 P0 테스트가 통과한다.
- smoke test가 모두 통과한다.
- 데이터 검증이 통과한다.
- 출시 차단 이슈가 0개다.
- major 이슈가 있으면 Product Owner가 known issue로 승인해야 한다.
- 모바일, 태블릿, 데스크톱 핵심 화면 QA가 완료되어야 한다.

### 11.2 이슈 심각도

| 심각도 | 설명 | 출시 여부 |
| --- | --- | --- |
| blocker | 핵심 기능 사용 불가 | 출시 불가 |
| critical | 잘못된 가격, 상세 오류, 모바일 핵심 UX 불가 | 출시 불가 |
| major | 일부 흐름 불편, 특정 조건 오류 | 승인 필요 |
| minor | 문구, 간격, 낮은 영향의 UI 문제 | 출시 가능 |
| trivial | 사소한 개선 사항 | 출시 가능 |

### 11.3 출시 차단 이슈

다음은 반드시 수정 후 배포한다.

- 검색이 동작하지 않음
- 원두 상세 접근 불가
- 가격 또는 100g당 가격이 잘못 표시됨
- 비교함 추가/제거 불가
- 판매처 이동 불가
- 모바일에서 검색 결과 또는 상세 사용 불가
- 필수 데이터 누락으로 화면 오류 발생
- production 빌드 실패

## 12. Production 배포 절차

1. staging QA 승인 기록을 확인한다.
2. production 배포 대상 commit을 확정한다.
3. production 환경 변수를 확인한다.
4. production 배포를 실행한다.
5. 배포 완료 URL을 확인한다.
6. production smoke test를 실행한다.
7. 이벤트 수집 여부를 확인한다.
8. 오류 로그를 확인한다.
9. 이상이 없으면 릴리즈 완료로 표시한다.
10. 이상이 있으면 롤백 기준에 따라 판단한다.

## 14. 롤백 기준

### 14.1 즉시 롤백

다음 상황이면 즉시 이전 안정 버전으로 롤백한다.

- 홈 화면 접속 불가
- 검색 API 또는 검색 기능 전체 장애
- 원두 상세 대부분 접근 불가
- 가격 정보가 대규모로 잘못 표시됨
- 판매처 링크가 대부분 잘못 연결됨
- production 빌드 또는 라우팅 장애
- 사용자 화면에서 반복적인 치명 오류 발생

### 14.2 수정 후 재배포

다음은 롤백보다 hotfix가 적합할 수 있다.

- 특정 원두 1-2개의 데이터 오류
- 일부 카테고리 설명 오타
- 특정 필터 조합에서만 빈 결과 오류
- 일부 이미지 깨짐
- 이벤트 일부 누락

### 14.3 롤백 절차

1. 장애 내용을 기록한다.
2. Release Owner가 롤백 여부를 결정한다.
3. 이전 안정 배포 버전으로 되돌린다.
4. production smoke test를 다시 실행한다.
5. 사용자 영향과 원인을 기록한다.
6. 수정 branch에서 원인을 해결한다.
7. staging 검증 후 재배포한다.

## 15. 출시 후 모니터링

### 15.1 모니터링 기간

### 15.2 기술 지표

| 지표 | 기준 |
| --- | --- |
| 홈 접속 오류율 | 급증 여부 확인 |

| JS runtime error | 급증 여부 확인 |
| 이벤트 전송 실패율 | 급증 여부 확인 |
| 판매처 이동 이벤트 | 기록 여부 확인 |

| 필터 사용률 | 필터 가치 확인 |
| 상세 진입률 | 검색 결과 품질 확인 |
| 비교함 추가 수 | 비교 기능 가치 확인 |

| 지표 | 목적 |
| --- | --- |
| stale 데이터 수 | 갱신 필요 원두 확인 |

## 16. Known Issues 관리

릴리즈 시점에 남길 수 있는 이슈는 다음 조건을 만족해야 한다.

- 핵심 흐름을 막지 않는다.
- 사용자에게 잘못된 가격이나 판매처를 제공하지 않는다.
- 명확한 해결 예정 시점이 있다.
- Product Owner가 승인했다.

Known issue 기록 형식:

```text
ID:
제목:
영향:
우회 방법:
해결 예정:
승인자:
```
